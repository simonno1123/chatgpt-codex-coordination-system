# Skill: Codex Task Writer

## 一、用途

本 skill 用于将 ChatGPT 的项目设计、审查意见、返工要求或用户目标，转化为 Codex 可以直接执行的结构化任务书。

它服务于 `agents/codex_execution_coordinator.md`，是 ChatGPT-Codex 协同作业机制中的任务生成组件。

本 skill 的目标不是让 Codex 自行规划项目，而是让 ChatGPT 将任务压缩为边界清晰、目标明确、可执行、可验收的小任务。

---

## 二、适用场景

本 skill 适用于以下场景：

1. 用户提出新的项目目标，需要拆解为 Codex 任务；
2. 需要创建新的 agent；
3. 需要创建新的 skill；
4. 需要创建模板文件；
5. 需要新增配置占位文件；
6. 需要让 Codex 修改指定文件；
7. 需要让 Codex 根据审查意见返工；
8. 需要让 Codex 根据报错日志局部修复；
9. 需要将法律业务逻辑落地为文档、规则或模板；
10. 需要生成一条可复制给 Codex 的完整任务书。

---

## 三、不适用场景

本 skill 不应用于：

1. 让 Codex 自行决定项目架构；
2. 让 Codex 自行选择法律业务规则；
3. 让 Codex 自行扩大任务范围；
4. 让 Codex 一次性重构整个项目；
5. 让 Codex 删除、迁移、重命名大量文件；
6. 让 Codex 编造外部 API、MCP、密钥、认证参数；
7. 让 Codex 未经授权修改依赖文件或核心配置文件；
8. 让 Codex 在没有验收标准的情况下自由发挥。

---

## 四、输入信息

使用本 skill 生成任务书前，应尽量明确以下信息：

1. 当前项目目标；
2. 当前项目阶段；
3. 已完成任务；
4. 本轮任务目标；
5. 允许新增的文件或目录；
6. 允许修改的文件或目录；
7. 禁止修改的文件或目录；
8. 是否允许运行测试；
9. 是否允许修改依赖；
10. 是否允许修改配置；
11. 是否存在法律业务边界；
12. 本轮任务的验收标准；
13. Codex 回报格式；
14. 遇到不确定事项时的 BLOCKED 规则。

如果上述关键信息缺失，应优先使用保守任务边界，不得默认授权 Codex 扩大修改范围。

---

## 五、输出目标

本 skill 应输出一份完整的 Codex 执行任务书。

任务书必须做到：

1. Codex 可以直接复制执行；
2. 任务目标单一；
3. 操作范围明确；
4. 禁止事项明确；
5. 具体要求可逐项检查；
6. 验收标准清晰；
7. BLOCKED 规则明确；
8. DONE 回报格式固定；
9. 不依赖 Codex 自行猜测；
10. 不把法律业务判断交给 Codex 决定。

---

## 六、任务拆分原则

生成 Codex 任务时，应优先采用小步任务。

### 1. 推荐任务粒度

推荐：

```text
只创建一个 agent 文件。
只创建一个 skill 文件。
只修改一个指定模板。
只新增一个配置占位说明。
只根据一个报错修复一个问题。
只运行一次指定测试并回报结果。
```

不推荐：

```text
把整个项目做好。
完整本土化所有模块。
优化全部代码。
把所有 agent 和 skill 一次性写完。
你自己看着改。
```

### 2. 单任务单目标

每个任务原则上只应有一个核心目标。

如果任务包含多个互相独立的目标，应拆成多个任务。

### 3. 先结构后内容

项目初期应优先创建：

1. 主控 agent；
2. 任务生成 skill；
3. 输出审查 skill；
4. 中断处理 skill；
5. 上下文压缩 skill；
6. 协作目录协议；
7. 具体法律业务 agent；
8. 具体法律业务 skill；
9. 输出模板；
10. 测试样例。

---

## 七、任务书标准结构

生成任务书时，应使用以下结构：

```markdown
# Codex 执行任务：[TASK_ID]

## 一、任务背景

说明当前项目背景、已完成内容和本任务来源。

## 二、任务目标

明确本轮任务只完成什么。

## 三、允许操作范围

列明允许新增、修改或查看的文件和目录。

## 四、禁止事项

列明 Codex 不得执行的操作。

## 五、具体要求

逐条列出本任务必须完成的内容。

## 六、验收标准

列明任务完成后应满足的条件。

## 七、BLOCKED 规则

说明遇到什么情况必须暂停并提问。

## 八、DONE 回报格式

规定 Codex 完成后必须如何回报。
```

---

## 八、任务背景写法

任务背景应包含：

1. 当前项目阶段；
2. 已完成任务；
3. 本任务与前序任务的关系；
4. 本任务不处理的内容；
5. 需要 Codex 严格遵守的边界。

示例：

```markdown
## 一、任务背景

当前项目已创建 `agents/codex_execution_coordinator.md`，用于作为 ChatGPT-Codex 协同作业的主控 agent。

本次任务只创建 `skills/codex_task_writer.md`，用于生成 Codex 任务书。

本任务不创建其他 skill，不创建 `.codex-coordination/`，不修改已有 agent，不处理 Git 问题。
```

---

## 九、任务目标写法

任务目标必须具体、单一、可验证。

推荐：

```markdown
## 二、任务目标

只创建以下文件：

- skills/codex_task_writer.md
```

不推荐：

```markdown
## 二、任务目标

完善整个协同系统。
```

---

## 十、允许操作范围写法

允许操作范围必须列明具体路径。

示例：

```markdown
## 三、允许操作范围

允许新增：

- skills/
- skills/codex_task_writer.md

不允许修改：

- agents/codex_execution_coordinator.md
- README.md
- package.json
- requirements.txt
- pyproject.toml
- 任何已有源码文件
```

如果任务需要修改已有文件，应明确列出：

```markdown
允许修改：

- skills/bank_flow_analysis.md
```

不得写成：

```markdown
可以修改相关文件。
```

---

## 十一、禁止事项写法

禁止事项应尽量具体，至少包括：

1. 不得删除已有文件；
2. 不得修改未授权文件；
3. 不得引入新依赖；
4. 不得擅自调整架构；
5. 不得编造外部服务配置；
6. 不得处理本任务之外的问题；
7. 遇到同名文件必须 BLOCKED；
8. 遇到项目结构冲突必须 BLOCKED。

---

## 十二、具体要求写法

具体要求应可逐项检查。

示例：

```markdown
## 五、具体要求

1. 创建 `skills/codex_task_writer.md`。
2. 文件必须包含用途、适用场景、输入、输出、任务书标准结构、禁止事项、验收标准。
3. 不得创建其他文件。
4. 不得修改已有文件。
```

---

## 十三、验收标准写法

验收标准必须让 ChatGPT 能够判断是否通过。

示例：

```markdown
## 六、验收标准

1. 成功创建指定文件。
2. 未修改其他文件。
3. 未删除任何文件。
4. 文件内容包含用途、输入、输出、任务书格式、BLOCKED 规则和 DONE 回报格式。
5. Codex 输出新增文件清单和验证结果。
```

---

## 十四、BLOCKED 规则写法

每个任务必须包含 BLOCKED 规则。

标准写法：

```markdown
## 七、BLOCKED 规则

如遇以下情况，暂停并输出 BLOCKED：

1. 目标文件已存在；
2. 项目中已有不同格式规范；
3. 完成任务需要修改未授权文件；
4. 当前目录结构与任务描述不一致；
5. Codex 无法判断是否应覆盖、迁移或重命名文件。
```

BLOCKED 回报格式：

```markdown
# Codex 执行中断问题

## 1. 当前任务

## 2. 已完成内容

## 3. 卡点描述

## 4. 涉及文件

## 5. 可选方案

### 方案 A

### 方案 B

### 方案 C

## 6. Codex 初步判断

## 7. 需要 ChatGPT 决策的问题
```

---

## 十五、DONE 回报格式写法

每个任务必须规定 DONE 回报格式。

标准写法：

```markdown
## 八、DONE 回报格式

完成后请输出：

1. 任务状态：DONE
2. 修改摘要：
3. 新增文件：
4. 修改文件：
5. 删除文件：
6. 运行命令：
7. 验证方式：
8. 测试结果：
9. 潜在风险：
10. 建议下一步：
```

---

## 十六、默认禁止修改文件

除非任务书明确授权，Codex 任务书中应默认禁止修改以下文件：

```text
package.json
package-lock.json
pnpm-lock.yaml
yarn.lock
pyproject.toml
requirements.txt
README.md
.env
.env.local
.gitignore
Dockerfile
docker-compose.yml
```

如果确需修改上述文件，任务书必须单独说明理由，并要求 Codex 完成后报告具体 diff。

---

## 十七、法律业务任务的特殊要求

当任务涉及中国民商事诉讼、民事执行、公司责任、股东责任、人格否认、债权人撤销权、银行流水、证据审查、北大法宝 MCP 等内容时，任务书必须增加法律业务边界。

应明确：

1. Codex 只能根据任务书落地结构；
2. Codex 不得自行编造法律依据；
3. Codex 不得自行编造案例；
4. Codex 不得自行扩展浙江地区裁判规则；
5. Codex 不得把未经确认的诉讼策略写成确定结论；
6. 不确定法律问题必须 BLOCKED；
7. 输出中必须保留风险、反方抗辩、补证清单等结构。

---

## 十八、返工任务写法

当 Codex 输出需要返工时，任务书应更严格。

返工任务必须包含：

1. 上一步问题；
2. 需要修改的具体文件；
3. 不允许修改的文件；
4. 需要删除或保留的内容；
5. 修改后的验收标准；
6. 不得顺手优化其他内容。

示例：

```markdown
# Codex 返工任务：REWORK_TASK_003

## 一、返工原因

上一步 `skills/bank_flow_analysis.md` 存在以下问题：

1. 只做金额汇总；
2. 未分析资金流向；
3. 未识别提现、自我转账、关联账户转移；
4. 未输出履行能力判断。

## 二、任务目标

只修改：

- skills/bank_flow_analysis.md

## 三、禁止事项

1. 不得修改其他文件；
2. 不得新增新 skill；
3. 不得删除原有风险提示结构；
4. 不得编造案例或法条。
```

---

## 十九、测试任务写法

当需要 Codex 运行测试时，应明确：

1. 运行什么命令；
2. 在什么目录运行；
3. 是否允许自动修复；
4. 修复范围是什么；
5. 如果失败如何回报。

示例：

````markdown
## 测试要求

请运行：

```bash
pytest
````

如果测试失败：

1. 先不要大范围修复；
2. 输出失败日志；
3. 判断失败是否与本任务有关；
4. 仅在本任务授权文件范围内修复；
5. 超出范围则进入 BLOCKED。
````

---

## 二十、输出风格要求

生成的 Codex 任务书应：

1. 直接可复制；
2. 指令清晰；
3. 少用抽象表述；
4. 不让 Codex 自由发挥；
5. 不默认授权；
6. 不含模糊措辞；
7. 不把业务判断交给 Codex；
8. 每条要求尽量可验收。

---

## 二十一、最低任务书模板

当信息不足时，应使用以下保守模板：

````markdown
# Codex 执行任务：[TASK_ID]

## 一、任务背景

当前任务由 ChatGPT 生成。由于项目信息有限，本任务采用最小修改范围原则。

## 二、任务目标

只完成以下目标：

- [明确目标]

## 三、允许操作范围

允许新增或修改：

- [具体路径]

## 四、禁止事项

1. 不得删除任何已有文件。
2. 不得修改未授权文件。
3. 不得引入新依赖。
4. 不得修改配置文件。
5. 不得处理本任务之外的问题。
6. 遇到不确定事项必须 BLOCKED。

## 五、具体要求

1. [要求一]
2. [要求二]
3. [要求三]

## 六、验收标准

1. 完成指定目标。
2. 未越权修改。
3. 未删除文件。
4. 输出验证结果。
5. 输出风险说明。

## 七、BLOCKED 规则

如遇同名文件、路径冲突、需要修改未授权文件或无法判断的情况，暂停并提问。

## 八、DONE 回报格式

完成后请输出：

1. 任务状态：DONE
2. 修改摘要：
3. 新增文件：
4. 修改文件：
5. 删除文件：
6. 验证方式：
7. 测试结果：
8. 潜在风险：
9. 建议下一步：
````

---

## 二十二、最终目标

本 skill 的最终目标，是让 ChatGPT 能持续、稳定、低歧义地向 Codex 下发任务，减少 Codex 无效探索、越权修改和返工成本。

任务书的核心不是“让 Codex 聪明地发挥”，而是让 Codex 在明确边界内完成可验收的执行动作。
