# Skill: Codex Output Reviewer

## 一、用途

本 skill 用于审查 Codex 完成任务后的输出结果，判断该任务是否可以验收、是否需要返工，或者是否因信息不足、结构冲突、权限边界不清而继续处于 BLOCKED 状态。

它服务于 `agents/codex_execution_coordinator.md`，是 ChatGPT-Codex 协同作业机制中的结果审查组件。

本 skill 的核心目标不是信任 Codex 的自述，而是根据任务书、文件清单、diff、测试结果、报错日志和风险说明，对 Codex 的执行结果进行结构化审查。

---

## 二、适用场景

本 skill 适用于以下场景：

1. Codex 回报任务状态为 DONE；
2. Codex 回报任务状态为 BLOCKED，但其中包含部分已完成内容；
3. Codex 输出新增文件清单；
4. Codex 输出修改文件清单；
5. Codex 输出删除文件清单；
6. Codex 输出测试结果；
7. Codex 输出报错日志；
8. Codex 输出潜在风险；
9. 用户需要判断是否进入下一任务；
10. 用户需要生成返工任务。

---

## 三、输入信息

使用本 skill 审查 Codex 输出时，应尽量取得以下信息：

1. 原始 Codex 任务书；
2. Codex 的任务状态；
3. 修改摘要；
4. 新增文件清单；
5. 修改文件清单；
6. 删除文件清单；
7. 运行命令；
8. 测试结果；
9. 验证方式；
10. 潜在风险；
11. Codex 建议下一步；
12. git diff 或等价的文件差异；
13. 关键文件内容摘要；
14. 报错日志；
15. artifact metadata，包括 `ARTIFACT TYPE`、`PRODUCER`、`TO`、`NEXT RECEIVER`、`MODE`、`PROJECT`、`AUTHORITY LIMIT`、`FORBIDDEN`、`OUTPUT`。

如果缺少 diff 或关键文件内容，应在审查结论中说明审查限制，不得在证据不足时作无保留验收。

---

## 四、审查目标

本 skill 应回答以下问题：

1. Codex 是否完成了本轮任务目标；
2. Codex 是否只在授权范围内操作；
3. Codex 是否修改了禁止修改的文件；
4. Codex 是否删除了任何文件；
5. Codex 是否引入了未授权依赖；
6. Codex 是否修改了核心配置；
7. Codex 是否擅自扩大任务范围；
8. Codex 是否伪造了接口、密钥、认证参数或外部服务配置；
9. Codex 是否错误处理了未授权领域规则或业务逻辑；
10. Codex 是否运行了必要测试；
11. 未运行测试是否合理；
12. 当前风险是否可接受；
13. 是否可以进入下一任务；
14. 是否需要返工；
15. 是否需要用户或 ChatGPT 进一步决策；
16. artifact metadata 是否完整；
17. `PRODUCER`、`ARTIFACT TYPE`、`NEXT RECEIVER` 和 `AUTHORITY LIMIT` 是否符合 ACOS 权限模型。

---

## 五、审查维度

### 1. 任务目标审查

检查 Codex 输出是否完成原任务目标。

应确认：

1. 目标文件是否创建；
2. 指定内容是否写入；
3. 指定修改是否完成；
4. 本任务不应处理的内容是否被擅自处理；
5. 任务是否被扩大或偏离。

### 2. 文件范围审查

检查 Codex 是否只操作了授权文件。

应重点关注：

1. 新增文件是否在允许范围内；
2. 修改文件是否在允许范围内；
3. 删除文件是否存在；
4. 是否修改依赖文件；
5. 是否修改 README；
6. 是否修改配置文件；
7. 是否修改 Git 相关文件；
8. 是否创建了未授权目录。

### 3. 内容质量审查

检查新增或修改内容是否符合任务书。

应重点关注：

1. Markdown 结构是否完整；
2. 是否保留核心章节；
3. 是否缺少 BLOCKED 规则；
4. 是否缺少 DONE 回报格式；
5. 是否缺少验收标准；
6. 是否出现空泛、不可执行表述；
7. 是否把判断责任错误交给 Codex；
8. 是否把不确定事项写成确定规则。

### 4. Domain-specific boundary review

涉及任何具体业务领域、外部知识体系、领域数据、专用工作流、专门工具或外部服务时，应重点检查：

1. 是否编造外部依据、示例数据、服务能力或领域结论；
2. 是否把未经确认的领域策略写成绝对结论；
3. 是否自行扩展地域、行业、组织或专用流程规则；
4. 是否遗漏风险提示、假设和待确认事项；
5. 是否遗漏后续验证清单；
6. 是否将未授权领域判断权交给 Codex；
7. 是否用泛泛表述替代结构化领域分析；
8. 是否越过任务书给定的领域边界。

### 5. Artifact routing and authority review

检查 Codex 输出是否符合 ACOS artifact routing 要求。

应重点关注：

1. 是否声明 `ARTIFACT TYPE`；
2. 是否声明 `PRODUCER`；
3. 是否声明 `TO`；
4. 是否声明 `NEXT RECEIVER`；
5. 是否声明 `MODE`、`PROJECT`、`AUTHORITY LIMIT`、`FORBIDDEN` 和 `OUTPUT`；
6. Codex 是否只产出 `RESULT` 或 `BLOCKED RESULT`；
7. Codex 是否错误产出 `REVIEW` 或 `DECISION`；
8. External Advisory Reviewer 是否仅限 `ADVISORY REVIEW`；
9. `NEXT RECEIVER` 是否位于输出最后，且与任务生命周期一致；
10. 是否存在自我验收、身份伪造或越权 artifact。

### 6. 测试与验证审查

检查 Codex 是否进行了适当验证。

应区分：

1. Markdown 文件创建任务：文件存在性检查通常可以接受；
2. 代码修改任务：原则上应运行相关测试；
3. 配置修改任务：应至少进行格式校验；
4. 依赖修改任务：应运行安装或构建验证；
5. 测试失败但与本任务无关：可记录风险；
6. 测试失败且与本任务有关：原则上不得 ACCEPTED。

### 7. 风险审查

应识别以下风险：

1. 无 diff 导致无法实质审查；
2. `.git` 不可用导致无法可靠比较改动；
3. Codex 只提供摘要，未提供关键文件内容；
4. 任务执行范围不透明；
5. 测试未运行；
6. 文件内容可能不完整；
7. 后续任务依赖当前文件质量；
8. 项目结构尚未标准化。

---

## 六、审查结论类型

审查结论只能使用以下三种之一：

```text
ACCEPTED
REWORK
BLOCKED
```

### 1. ACCEPTED

仅在以下条件基本满足时使用：

1. 完成任务目标；
2. 未越权修改；
3. 未删除文件；
4. 未引入未授权依赖；
5. 未处理禁止事项；
6. 内容结构符合任务书；
7. 风险可控；
8. 未发现需要立即返工的问题。

如果缺少 diff 或完整文件内容，但 Codex 回报可信且任务简单，可以使用：

```text
条件验收：ACCEPTED
```

但必须说明审查限制。

### 2. REWORK

出现以下情况时使用：

1. 任务部分完成但内容不完整；
2. 目标文件创建了但核心章节缺失；
3. 文件内容偏离任务书；
4. 出现轻微越权修改但可修复；
5. Markdown 结构混乱，影响后续使用；
6. 领域或业务边界表达不充分；
7. BLOCKED 或 DONE 机制缺失；
8. 需要局部补充、删除或重写。

### 3. BLOCKED

出现以下情况时使用：

1. 缺少关键信息，无法判断是否通过；
2. Codex 修改了未授权文件，但未提供 diff；
3. Codex 删除了文件；
4. Codex 修改了依赖或配置文件；
5. 测试失败且原因不明；
6. 项目结构冲突；
7. 需要用户决策；
8. 涉及外部服务配置但参数不明；
9. 涉及未授权领域规则或业务逻辑但需要 ChatGPT 或用户确认；
10. 当前任务无法继续审查。

---

## 七、审查输出格式

审查 Codex DONE 结果时，使用以下格式：

```markdown
# 审查结论：[ACCEPTED / REWORK / BLOCKED]

## 一、结论

[明确说明是否通过、是否条件验收、是否需要返工或阻塞]

## 二、审查依据

1. 原任务目标：
2. Codex 回报内容：
3. 文件操作范围：
4. 验证或测试结果：
5. 风险说明：

## 三、通过项

1. 
2. 
3. 

## 四、问题或风险

1. 
2. 
3. 

## 五、处理意见

[如 ACCEPTED，说明下一步。]
[如 REWORK，给出返工要求。]
[如 BLOCKED，列出需要补充的信息或决策。]

## 六、下一步任务建议

[给出下一步 Codex 任务方向。]

## NEXT RECEIVER

[ChatGPT Review / Codex Executor / User Decision / External Advisory Reviewer / None]

## Reason

[说明为什么交给该接收方]
```

---

## 八、返工指令生成规则

如果结论为 REWORK，应生成明确返工任务。

返工指令必须包含：

1. 返工原因；
2. 只允许修改的文件；
3. 禁止修改的文件；
4. 具体修改要求；
5. 不得顺手优化；
6. 验收标准；
7. DONE 回报格式。

返工指令示例：

```markdown
# Codex 返工任务：REWORK_TASK_003

## 一、返工原因

上一步输出存在以下问题：

1. 缺少 BLOCKED 规则；
2. 缺少 DONE 回报格式；
3. 擅自增加了未要求的外部服务配置。

## 二、任务目标

只修改：

- skills/codex_output_reviewer.md

## 三、允许操作范围

允许修改：

- skills/codex_output_reviewer.md

## 四、禁止事项

1. 不得修改其他文件；
2. 不得新增文件；
3. 不得删除文件；
4. 不得引入依赖；
5. 不得处理 `.git` 问题。

## 五、具体要求

1. 补充 BLOCKED 规则；
2. 补充 DONE 回报格式；
3. 删除未经授权的外部服务配置内容。

## 六、验收标准

1. 指定文件完成修正；
2. 未修改其他文件；
3. 未删除文件；
4. 输出修正摘要。
```

---

## 九、BLOCKED 处理规则

如果结论为 BLOCKED，应说明阻塞原因，并明确下一步需要谁提供什么。

BLOCKED 输出应包含：

1. 阻塞原因；
2. 缺失信息；
3. 当前不得继续的事项；
4. 需要用户提供的信息；
5. 需要 Codex 补充的信息；
6. 是否需要读取文件内容；
7. 是否需要重新运行验证命令。

示例：

```markdown
# 审查结论：BLOCKED

## 一、阻塞原因

Codex 声称修改了多个文件，但未提供修改文件清单和 diff，无法判断是否越权。

## 二、需要补充的信息

请 Codex 补充：

1. 新增文件清单；
2. 修改文件清单；
3. 删除文件清单；
4. 每个文件的内容摘要；
5. 如 git 可用，提供 git diff；如不可用，说明原因。

## 三、暂停事项

在补充上述信息前，不得进入下一任务。
```

---

## 十、条件验收规则

当任务较简单、Codex 回报完整，但缺少 diff 或完整文件内容时，可以进行条件验收。

条件验收必须写明：

1. 审查依据有限；
2. 未审查完整文件内容；
3. 未审查真实 diff；
4. 仅基于 Codex 回报和任务简单性通过；
5. 后续发现问题可返工。

示例：

```markdown
# 审查结论：条件验收：ACCEPTED

## 一、结论

本任务可条件验收。

## 二、审查限制

当前仅收到 Codex 执行摘要，未收到完整文件内容或 diff，因此无法进行实质内容逐行审查。

## 三、通过理由

1. 任务目标单一；
2. Codex 回报显示只新增指定文件；
3. 未修改或删除其他文件；
4. 验证方式与 Markdown 文件创建任务相匹配。

## 四、保留风险

如后续使用中发现文件内容缺漏，应通过 REWORK 任务局部修正。
```

---

## 十一、不同任务类型的验收重点

### 1. 创建 Markdown 文件

重点审查：

1. 是否只创建指定文件；
2. 文件路径是否正确；
3. 章节是否完整；
4. 是否未修改其他文件；
5. 未运行项目测试是否合理。

### 2. 修改 Markdown 文件

重点审查：

1. 是否只修改指定文件；
2. 是否保留原有核心结构；
3. 是否补充了指定内容；
4. 是否删除了不应删除的部分；
5. 是否存在过度发挥。

### 3. 创建代码文件

重点审查：

1. 文件路径；
2. 代码是否可运行；
3. 是否符合现有架构；
4. 是否有必要测试；
5. 是否存在安全问题；
6. 是否引入未授权依赖。

### 4. 修改代码文件

重点审查：

1. 是否只改授权文件；
2. 是否解决指定问题；
3. 是否破坏其他逻辑；
4. 是否运行相关测试；
5. 是否有回滚风险。

### 5. 配置文件任务

重点审查：

1. 是否编造认证信息；
2. 是否使用占位符；
3. 是否明确用户需自行配置；
4. 是否修改真实 `.env`；
5. 是否影响现有运行。

### 6. 外部工具或服务占位任务

重点审查：

1. 是否只是占位说明；
2. 是否未编造密钥；
3. 是否未编造不可确认 API；
4. 是否说明需要用户提供真实配置；
5. 是否与项目现有外部工具或服务结构兼容。

### 7. Domain-specific agent / skill 任务

重点审查：

1. 是否结构化表达已授权领域逻辑；
2. 是否区分事实、依据、假设、风险、待确认事项和验证清单；
3. 是否避免编造外部依据、示例数据或领域规则；
4. 是否保留反向风险或替代解释；
5. 是否强调材料不足时的结论边界；
6. 是否符合任务书明确授权的领域场景。

---

## 十二、默认高风险信号

出现以下情况时，原则上不得直接 ACCEPTED：

1. Codex 修改了未授权文件；
2. Codex 删除了文件；
3. Codex 修改了依赖文件；
4. Codex 修改了 `.env`；
5. Codex 修改了 Git 配置；
6. Codex 自动提交或部署；
7. Codex 编造了外部 API 配置；
8. Codex 处理了任务之外的问题；
9. Codex 没有提供文件清单；
10. Codex 没有说明测试结果；
11. Codex 对未授权领域问题作出未经确认的确定结论；
12. Codex 在 BLOCKED 条件下继续执行；
13. Codex 输出缺少 `NEXT RECEIVER`；
14. Codex 输出 `REVIEW` 或 `DECISION`；
15. Codex 将输出路由给自己验收；
16. Codex 伪造其他 producer 身份。

---

## 十三、默认可接受风险

以下情况可以记录风险但不必阻塞：

1. Markdown 文件创建任务未运行项目测试；
2. `.git` 不可用但本轮未依赖 diff；
3. Codex 只做了文件存在性检查；
4. 任务简单且只新增单个文件；
5. 文件内容后续可通过 REWORK 局部修正；
6. 当前阶段尚未建立完整协作目录协议。

---

## 十四、输出风格要求

审查意见应当：

1. 直接给出结论；
2. 明确审查限制；
3. 区分已确认事实和 Codex 自述；
4. 不夸大确定性；
5. 不在材料不足时无保留验收；
6. 如果返工，给出可复制给 Codex 的返工任务；
7. 如果通过，给出下一步任务建议；
8. 保持任务边界清楚。

---

## 十五、最终目标

本 skill 的最终目标，是让 ChatGPT 能够稳定审查 Codex 的执行结果，减少越权修改、遗漏验证、错误验收和反复返工。

审查的核心不是“相信 Codex 已经完成”，而是通过任务书、文件范围、验证结果和风险说明，判断该任务能否进入下一阶段。
